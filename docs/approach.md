# Approach Document — TickerWire-ITC Assistant

## Problem Decomposition

### The Core Pain Point
A 6-analyst desk spending 25–40 min per lookup query isn't a knowledge problem — it's a retrieval and trust problem. Journalists need answers *fast* and *verifiable* (wrong year = story retraction). So the system must: (a) find the right passage, (b) cite it precisely, (c) know when it doesn't know.

### Decomposition into Sub-Problems

**1. Data Layer**
ITC's annual reports are dense, mixed-format PDFs: narrative prose, financial tables, segment breakdowns, auditor notes. Tables are the hardest — PDF parsers often mangle multi-column tables. Decision: use `pdfplumber` for table extraction (preserves cell structure) and `PyMuPDF` for text flow, then merge.

Chunking strategy matters more than people acknowledge. Fixed-size chunking splits tables mid-row. Solution: detect table boundaries, keep tables as atomic chunks; apply sliding-window chunking to prose sections (512 tokens, 64-token overlap).

**2. Retrieval Layer**
Two failure modes to guard against:
- *Semantic miss*: "return on capital" doesn't match "ROCE" without semantic understanding → dense retrieval
- *Exact miss*: "FY2024 revenue" needs to match "2023-24" exactly → keyword/BM25

Hybrid with RRF combines both. Re-ranking with Cohere's cross-encoder then adds query-passage interaction that neither bi-encoder captures.

HyDE addresses the asymmetry between short journalistic queries and long report passages. A journalist asks: *"What was ITC's cigarette segment EBIT margin in FY24?"* — 12 words. The relevant passage is 200 words. HyDE generates a plausible answer paragraph (embedding target), then retrieves passages similar to *that* — dramatically improving recall on specific financial queries.

**3. Agent Layer**
A naive RAG always retrieves and answers. But:
- "What will ITC's revenue be in FY26?" should be refused (speculation)
- "ITC's revenue" without a year should trigger clarification (4 fiscal years in corpus)
- "Compare ITC's FY22 vs FY25 EBIT" needs multi-hop retrieval (retrieve twice)

A lightweight router (50-token structured output from GPT-4o-mini) classifies the query into one of four actions before any expensive retrieval. This avoids wasting retrieval budget on refusals and makes the system behavior predictable and auditable.

**4. MCP Tool**
The `get_financial_kpi` tool serves as a structured, reliable fallback for exact number lookups. When the agent routes to `direct_answer` for KPI-type queries, it calls this tool rather than relying on free-text retrieval — ensuring numbers are exactly right. The tool queries the indexed metadata store (not free-text search), so it's deterministic.

**5. Streaming & Latency**
The p95 ≤ 5s target with first-token < 1.5s requires parallelism:
- Route decision and HyDE expansion happen concurrently
- Dense and BM25 retrieval run concurrently (asyncio.gather)
- Reranking is the only sequential step before generation
- Generator streams tokens immediately

**6. Evaluation**
Built 50 Q&A pairs manually by reading the reports, covering: segment financials, year-over-year changes, ratios, narrative facts, and out-of-corpus traps. Ground truth includes both the correct answer and the specific chunk(s) that should be retrieved. This enables both retrieval metrics (Recall@5, nDCG@10) and generation metrics (correctness, faithfulness).

## What I Ruled Out and Why

### LangChain / LlamaIndex
Both frameworks abstract away the exact behavior I need to control (retry logic, cost guards, streaming order). For a system where correctness and auditability matter more than development speed, plain Python + well-chosen libraries is more debuggable. Framework magic creates invisible failure modes.

### Vector DBs with Managed Infrastructure (Pinecone, Weaviate, Qdrant Cloud)
Adds an external dependency, API keys, and network latency for a corpus of ~4 PDFs (~40K chunks). ChromaDB runs in-process, persists to disk, and is fast enough. Swap-out is easy if scale demands it.

### Claude / Anthropic API as Primary LLM
GPT-4o-mini has slightly better structured output reliability (JSON mode) at lower cost for the routing step. For generation, both are comparable; GPT-4o-mini wins on cost. If API diversity is a concern, the generator is abstracted behind an interface — swapping is 5 lines.

### Elasticsearch for BM25
Docker dependency, resource overhead, ops burden — all for BM25 on 4 PDFs. `rank_bm25` is a pure Python implementation with zero infra requirements and equivalent quality at this scale.

### Contextual Retrieval (Anthropic's technique)
This prepends each chunk with a context summary generated by a language model during ingestion. It's genuinely effective, but costs ~$0.10–0.20 to pre-process the entire corpus and requires re-ingestion whenever the model changes. HyDE achieves similar recall improvement at query time with no ingestion cost. At scale (100s of PDFs), contextual retrieval becomes more attractive.

### Query Rewriting
Tested in preliminary experiments. Added ~400ms latency. Recall@5 improved by 0.02 — not worth the cost and latency budget at current scale.

## Key Risks and Mitigations

| Risk | Mitigation |
|------|-----------|
| Table extraction errors (PDF mangling) | Validate table chunks against known KPIs during ingestion; flag anomalies |
| HyDE hallucination affecting retrieval | HyDE expansion is logged; if retrieved chunks don't score above threshold, fall back to raw query |
| Cost overrun on long queries | Cost guard: abort if estimated prompt > 6000 tokens, return partial answer with note |
| Year attribution errors | Chunks tagged with fiscal year at ingestion; citations include year metadata |
| Out-of-corpus hallucination | Refuse action in agent router; generator prompt instructs to only use provided context |
